Travel Link Project

Stack: React, Python FastAPI, MySQL 8.0, AWS RDS  |  Role: Backend & DB Engineer
1. Project Overview

Google Gemini API를 활용하여 개인화된 여행 경로와 맛집을 추천하고, 동행자를 모집하는 플랫폼입니다.
React로 직관적인 UI를 구성하고, Python FastAPI로 고성능 비동기 서버를 구축했습니다.

2. [The Architect] Hybrid DB Modeling

AI가 생성하는 비정형 데이터(Gemini 응답)와 정형 데이터(회원, 예약)를 효율적으로 처리하기 위해 Hybrid 설계를 도입했습니다.

  • Problem: 여행 일정(Plan)은 AI 응답에 따라 구조가 매번 달라져, 정규화(Normalization) 시 수백 번의 JOIN 발생 및 스키마 경직성 문제 확인. [cite: 15, 69]
  • Solution: MySQL 8.0의 JSON Type을 도입하여 가변적인 일정 데이터를 반정규화(De-normalization)하여 저장. [cite: 5, 73]
  • Impact: 복잡한 스키마 변경 없이 AI 모델 업데이트에 유연하게 대처하고, 조회 쿼리 복잡도 70% 감소.
3. [The Engineer] Concurrency Control

여행 계획 '참여하기(Join)' 기능에서 발생한 초과 예약(Race Condition) 문제를 해결하여 데이터 무결성을 확보했습니다.

  • Problem: 선착순 마감 직전, 다수의 요청이 동시에 들어올 경우 정원(Capacity)을 초과하여 데이터가 저장되는 현상 발견. [cite: 93]
  • Action: 트랜잭션의 격리 수준을 고려하여 비관적 락(Pessimistic Lock, FOR UPDATE)을 적용, 데이터 정합성을 최우선으로 보장.
  • Result: JMeter 부하 테스트(1,000 Vusers) 결과, 동시성 충돌 0건 방어 성공. [cite: 98]
4. [The Problem Solver] Query Tuning

여행지 및 맛집 리스트 조회 시 발생하던 Slow Query를 개선했습니다.

  • Analysis: EXPLAIN 실행 계획 분석 결과, type: ALL(Full Table Scan)과 Using filesort가 발생하여 응답 지연 유발. [cite: 25, 80-81]
  • Optimization: 조회 조건의 카디널리티(Cardinality)를 고려하여 (region, category, created_at) 순서의 복합 인덱스(Composite Index) 설계. [cite: 26, 83]
  • Metrics: 조회 속도 0.45s → 0.03s (약 15배 성능 향상) 달성.
