Travel Link Project

Stack: React, Python FastAPI, MySQL 8.0, AWS RDS  |  Role: Backend & DB Engineer
1. [The Architect] 요구사항 분석 및 설계 의도

비즈니스 요구사항: 사용자의 취향에 따라 매번 달라지는 여행 코스(비정형 데이터)와 회원/예약 정보(정형 데이터)를 효율적으로 저장해야 함.

  • 데이터 모델링 전략 (Hybrid Design): 정규화(Normalization)를 기본으로 하되, 비정형 데이터 처리를 위해 반정규화를 혼합. [cite: 6]
  • 설계 의도: 여행 세부 일정은 수정이 잦고 구조가 가변적이므로, 별도 테이블 분리 시 수백 번의 JOIN 발생 우려. → MySQL JSON Type을 도입하여 유연성 확보. [cite: 17]
2. Entity Relationship Diagram (ERD)

* 여행 계획(Plans)과 세부 일정(Details)의 1:N 관계 및 JSON 컬럼 활용 구조

3. [The Problem Solver] 성능 최적화 및 튜닝

메인 리스트 조회 시 발생하던 Slow Query (3.8s → 0.03s) 개선 사례입니다. [cite: 22, 27]

  • 문제(Problem): WHERE region = ? AND status = ? ORDER BY created_at 쿼리 실행 시 Full Table Scan 및 FileSort 발생 확인. [cite: 25, 81]
  • 해결(Solution): 카디널리티(Cardinality)가 높은 순서대로 (region, status, created_at) 복합 인덱스(Composite Index) 설계 및 적용. [cite: 83]
  • 결과(Result): 쿼리 실행 시간 약 120배 단축 및 DB CPU 부하 40% 감소.
4. [The Engineer] 동시성 제어

선착순 '참여하기' 기능에서 발생한 초과 예약(Race Condition) 방어.

  • 상황: 마지막 1자리 남았을 때 동시 요청 시 정원 초과 저장 문제. [cite: 93]
  • 해결: 트랜잭션 격리 수준을 고려하여 비관적 락(Pessimistic Lock, FOR UPDATE) 적용. [cite: 95]
  • 검증: JMeter 부하 테스트 결과 정합성 100% 유지. [cite: 99]
